# OMR 表格边界修正与 L 角模板等价设计

日期：2026-07-10

## 背景

`docs/answer-card-miniprogram-omr-comparison.md` 基于较早代码指出小程序在实时取景、输入尺度、L 角模板和纸张弯曲修正方面优于本项目。当前 `main` 已经通过共享 `ViewPort`、实时引导框和固定 16×16 单格采样解决了前两项，因此本轮只处理仍然成立且能直接提高识别准确性的差距：

1. `CornerAnchorMatcher` 的大图/小图 L 角细化点被错误地再次乘以主尺度，和小程序的 22 点模板不等价。
2. 当前答案格和考号格只由四角 homography 投影；纸张弯曲、复印缩放或局部打印偏移不能由单一平面模型表示。

小程序会沿预计的左右表格边界搜索真实亮暗跳变，并用逐行边界生成网格。直接替换本项目的 homography 会破坏动态题量、隐藏表头和编码角标路径，因此本设计只在证据可靠时叠加局部横向修正。

## 目标

- 四种 L 角、两种图像尺度的 22 个模板点和小程序坐标逐项一致。
- 默认编码角和 legacy 路径都能利用真实表格竖边修正答案格与考号格的横向位置。
- 平整纸、边界模糊、反光或错误边线候选不得因修正器产生新的错误结果。
- 修正器没有可靠证据时，输出必须与当前 homography 投影完全一致。
- 保留固定 16×16 单格采样、原始信息量门、质量门、必填格校验和 3/4 完整答案共识。
- `omr-core` 保持纯 JVM，不引入 Android、AndroidX 或 OpenCV。

## 非目标

- 不改变编码角标格式、解码容错或至少三个唯一角标的要求。
- 不复刻小程序的弱多帧确认、五秒内保留高分或现场数据上传。
- 不用边界修正放宽必读格完整性、曝光、清晰度或最小信息量门。
- 不对竖直方向做无证据的自由形变；本轮聚焦小程序已证明有效的左右表格边界修正。
- 不把 legacy L 角设为默认角标模式。

## 方案比较

### 方案 A：完整替换为小程序网格

按小程序的非均匀行列比例、左右边界、圆弧候选和最终转置点阵重建所有格子。它最接近固定小程序卡，但本项目模板由 `TemplateGeometry` 动态生成，不同题量、题型和隐藏表头并不共享小程序的固定 `lieshu/lieLeft` 结构。替换生产投影会扩大错误成功风险，不采用。

### 方案 B：homography 加可靠边界残差

保留 `AndroidPaperCoordinateProjector` 的模板到图像 homography。对每个实际需要读取的格子，先取得其左右边界的预计图像位置，再在有限横向窗口内寻找与“外亮内暗”方向一致的竖边。由同一表格区的可靠观测拟合随纵向变化的左右残差，修正格子四角的横坐标。证据不足时逐区或整帧回退。该方案兼容动态模板并能吸收纸张弯曲和局部打印偏移，采用此方案。

### 方案 C：只修 L 角模板

改动最小，但只改善 legacy 找卡，无法改善默认编码卡的弯纸错格，不足以满足整体识别准确性目标。

## 架构

### 1. 等价 L 角模板

新增内部对象 `MiniProgramCornerTemplate`，集中返回 `List<CornerTemplatePoint>`。坐标直接保存为小程序的最终像素偏移，不再把主点和细化点混在一组后统一缩放：

- 大图主尺度为 6，细化偏移为 2、4；
- 小图主尺度为 3，细化偏移为 1、2；
- LU、LD、RU、RD 每种严格为 22 个唯一的 `(rowOffset, columnOffset, expectedClass)` 点。

`CornerAnchorMatcher` 的模板扫描和 trace 评分都调用该对象。对象保持 `internal`，让纯 JVM 单测可以直接验证坐标，不使用反射，也不增加测试专用生产 API。

### 2. 边界观测器

新增 `TableBoundaryRefiner`，输入：

- 灰度 `MiniProgramFrame`；
- homography 投影后的 `AndroidPaperProjectedCells`；
- 当前 `AndroidPaperTemplateLayout`。

小程序卡使用共享表格边线，当前 Android 模板则为每个答案选项和考号数字绘制独立矩形。因此修正器不能只搜索答案区最外侧边界，而要按当前真实印刷结构处理：答案格按 `questionIndex` 分组，考号格按 `digitIndex` 分组。每组从实际存在的 projected cell 推导：

- 每个矩形的预计左、右投影边；
- 投影边的方向和长度；
- 中位格宽，用于限定搜索半径和最大修正量。

沿投影边从高度 10% 到 90% 均匀取样，并对候选横向偏移累计亮度差。左边使用左侧外部条带与边线比较，右边使用右侧外部条带与边线比较；不使用内部条带，避免真实涂黑让边线失去对比。搜索范围限制为预计边界左右不超过中位格宽的 30%，避免吸附到相邻选项线。每个候选记录偏移、外边对比度和有效采样数。

### 3. 可靠性与平滑

单条矩形边观测只有同时满足以下条件才有效：

- 对比度至少 24 灰度级；
- 有效边线采样不少于计划样本的 70%；
- 偏移绝对值不超过中位格宽的 30%；
- 最佳候选比第二候选至少高 6 灰度级，排除重复竖线歧义。

同一题目行或考号行把有效边观测表示为 `(expectedColumn, observedOffset)`，先用偏移中位数和 MAD 去除离群点，再对剩余点拟合一次函数。一个分组启用修正还必须满足：

- 至少三个有效边观测，且至少来自两个不同矩形；
- 有效观测覆盖该分组预计横向跨度至少 45%；
- 拟合后残差均方根不超过 `max(1.5 px, 中位格宽 × 0.12)`；
- 拟合在分组左右端预测的偏移绝对值都不超过中位格宽的 30%；
- 修正后每个矩形右边界仍在左边界右侧，宽度位于原预计宽度的 70%～130%。

一次函数允许同一局部行内同时存在平移和轻微横向缩放。每个题目行独立拟合，使同一列在不同纵向位置可以获得不同残差，从而表达纸张缓慢弯曲；不做跨行高阶圆弧外推，避免稀疏点造成振荡。

### 4. 格子修正

`AndroidPaperProjectedCellBuilder.build` 先生成原始 projected cells，再调用修正器。每个格子的左上/左下角使用拟合函数在左边预计列的残差，右上/右下角使用右边预计列的残差。

修正只改变 `column`，保留 homography 给出的 `row`。答案题目行和考号行分别计算，避免不同印刷区域互相牵引。输出 `AndroidPaperProjectedCells.debugInfo` 记录成功修正的分组数、回退分组数、有效观测数、最大残差和主要回退原因。

### 5. 失败闭合与回退

任何异常、非有限坐标、低置信观测、几何交叉或修正后源信息量不足都会让对应表格区回退到原 projected cells。修正器本身不返回扫描成功或失败，只返回“原格子”或“经验证的格子”。后续现有的：

- `AndroidRequiredCellValidator`；
- `MiniProgramCellSampler.validateSource`；
- answer/admission reader；
- 评分与共识；

继续作为最终安全门。边界修正不能绕过这些检查。

## 数据流

```text
角标定位
  -> 模板到图像 homography
  -> 原始 projected cells
  -> 按题目行/考号行搜索各矩形真实竖边
  -> 可靠性、歧义、覆盖与平滑检查
       -> 可靠：按纵向边界残差修正 cells
       -> 不可靠：原 cells 原样回退
  -> 必读格完整性与源信息量验证
  -> 固定 16x16 单格采样与判格
  -> 评分与完整答案多帧共识
```

## 测试策略

### L 角模板

- 四种角 × 大小两种分支，逐项断言 22 个最终坐标与 JS 数组一致。
- 断言每个模板 22 点唯一，防止当前重复细化点回归。
- 在带轻微模糊/抗锯齿的合成 L 角上验证正确锚点仍能定位。

### 边界观测单测

- 平直矩形边的残差为零或不启用，输出格子与原投影一致。
- 不同题目行按纵向缓慢弯曲时分别恢复已知局部残差，误差不超过 1.5 px。
- 只有单个矩形、低对比、重复竖线、强反光和离群候选时按分组回退。
- 修正不得造成格子交叉、面积低于原始 70% 或超出帧边界。

### 引擎集成回归

- 对编码角和 legacy 各生成平整、透视和弯曲卡；弯曲样本在原 homography 下应出现可复现的错格或拒绝，修正后答案、考号和得分正确。
- 现有 `DesktopWechatImageScanTest` 的 5 张真实图片在原始分辨率和 1280 宽版本全部保持预期答案、考号和分数。
- 现有 coded/solid/L-bracket、close crop、required cell、cell sampler 测试全部通过。
- 新增 debug 断言证明弯曲样本实际启用了修正，防止测试只靠原路径偶然通过。

## 验收标准

- L 角模板坐标与小程序源码完全等价。
- 至少一个编码角和一个 legacy 合成弯曲样本由失败或错读变为完全正确。
- 所有平整与透视回归保持完全正确，不出现新的错误成功。
- 低置信边界样本明确记录回退，且其 projected cells 与未修正版本逐点一致。
- Fedora 上 `:omr-core:test`、项目 `test` 和 `assembleDebug` 通过；若 SSH 临时不可用，目标保持未完成并继续重试，不能用静态检查代替。

## 实施边界

1. 先以坐标测试修复 L 角模板。
2. 再为边界观测与置信回退写纯 JVM 失败测试。
3. 实现独立修正器并保持原 projected cells API。
4. 接入 projected cell builder，补编码角与 legacy 弯曲集成测试。
5. 跑真实图片、全量 JVM、Android 单测和 APK 构建。
