# 相机动态识别后续问题

记录日期：2026-07-10
状态：P0 代码与自动化回归已完成；P0-3 的两台真机标定待设备接入

本轮已完成 P0 的扫描安全边界、相机状态/质量双门控、择优帧窗口、UI 重拍提示和自动化回归。P1 的相机能力自适应、统一 `ViewPort` 与更高分辨率评估继续保留为独立后续工作。

### 2026-07-10 实施结果

- 默认扫描使用 `CODED_ONLY`，扫描页可显式切换“旧版”兼容模式；编码失败不再回退到实心/L 角。
- 旧实心/L 角定位记录最佳和次佳候选，同一印刷笔画内的 trace 抖动归为同一定位簇，跨定位簇分差不足或 `outer`/`innerRight` 映射冲突时返回 `LEGACY_ANCHOR_AMBIGUOUS`，且 `score=null`。
- CameraX 捕获结果已接入 AF/AE、曝光时间和 ISO；分析顺序为稳定性 → AF/AE → 整帧质量 → 450ms 择优帧 → 角标定位 → 卡片 ROI 质量 → 读格/评分 → 多帧共识。
- 卡片 ROI 清晰度门槛由现有真图扰动集标定：清晰帧归一化 Laplacian 为 538～2510，轻度 `blur1` 为 226，已产生不稳定多涂的 `blur2` 为 68，因此当前保守拒绝门槛设为 100。
- Fedora 已通过 `:omr-core:test`、`:app:testDebugUnitTest`、仓库级 `test` 和 `assembleDebug`。强过曝与 `blur2` 回归均断言 `success=false`、`score=null`。
- Fedora 的 `adb devices -l` 当前没有连接设备，因此“两台不同 Android 设备”的 AF/AE 状态兼容性、阈值分布和实际锁定延迟仍需真机补验，不能仅由桌面扰动集宣称完成。

## 1. 必须先落实的安全边界

相机扫描的首要目标是避免“看起来成功但分数错误”。只有帧质量合格、角标路径明确、定位候选唯一且几何校验通过的结果才能进入读格、评分和多帧共识。任一条件不满足时都必须返回可区分的失败原因并提示用户重拍，不得生成分数，也不得让失败帧参与共识。

### 编码角标路径

- 新版应用打印的答题卡使用编码角标。扫描会话应明确传入 `CODED_ONLY` 或 `LEGACY` 兼容模式，不能仅凭一次检测失败在三条路径之间自动猜测。
- `CODED_ONLY` 模式下，编码角标无法可靠解码、ID 重复、方向冲突、重投影误差超限或几何校验失败时，直接提示“角标不清晰，请对准后重拍”，不得回退到实心角标或 L 角路径后继续评分。
- 允许由 3 个唯一编码角标推断第 4 个，但必须同时满足解码误码、唯一性、方向、重投影和卡片几何门槛；少于 3 个唯一编码角标不评分。
- 过渡期若尚未增加显式扫描模式，只要出现可信编码角标证据，就必须锁定编码路径；不得把同一帧再解释成旧实心/L 角卡。当前实现仅在检测到至少 2 个编码 ID 时阻止回退，仍不足以覆盖检测到 0～1 个 ID 的新版坏帧。

### 旧实心角标与 L 角路径

- 旧实心角标和 L 角只能在显式 `LEGACY` 兼容模式下启用，且兼容模式也必须失败闭合，不能以“总能给出一个分数”为目标。
- `SolidCornerMarkerDetector` 与 `CornerAnchorMatcher` 必须保留最佳和次佳四角组合的分数、来源和坐标。若两个候选都通过基础校验，但分差不足，或映射后的关键答题格位置不一致，则返回“定位有歧义，请重拍”，禁止输出分数。
- L 角路径还要比较 `outer` 与 `innerRight` 两套参考。若两者证据接近却映射到不同答题格、准考证号或分数，不得按候选顺序静默选一个。
- 不能用多帧多数投票掩盖单帧定位歧义；只有每一帧先独立通过候选唯一性门控，才能进入共识。

### 结果契约与验收标准

- 为拒绝原因使用稳定枚举，例如 `WAIT_FOCUS`、`WAIT_EXPOSURE`、`RETAKE_BLUR`、`RETAKE_EXPOSURE`、`RETAKE_CODED_MARKERS`、`LEGACY_ANCHOR_AMBIGUOUS`，UI 再映射为中文提示；不要依赖英文 `failureReason` 文本判断流程。
- 所有拒绝结果都必须满足 `success=false`、`score=null`，并记录 `anchorMode`、`anchorPath`、最佳/次佳候选分数、分差和最终拒绝原因。
- 编码路径失败不会落入 `solid-marker` 或 `l-bracket`；旧实心/L 角候选有分歧时，不会进入读格、评分或共识。

## 2. 已实现相机帧质量门控

### 实现现状

- `AndroidOmrImageAnalyzer` 已在完整 OMR 前执行设备稳定性、AF/AE 和整帧质量门控。
- `AndroidOmrEngine` 在定位后对卡片 ROI 再做清晰度、曝光裁剪和动态范围复核；拒绝结果不进入读格、评分或共识。
- 强过曝以及已出现不稳定答案的 `blur2` 真图扰动均回归为可见失败，`score=null`。

### 已实现方案

- 将门控拆成两层：在完整 OMR 前用低成本指标拒绝明显坏帧；定位成功后只在卡片四边形及答题区 ROI 内复核清晰度、曝光裁剪比例、局部对比度、卡片面积和重投影误差，然后才允许读格和评分。
- 新增纯 JVM 的 `FrameQualityEvaluator`，返回结构化指标、是否接纳和拒绝原因。Laplacian 方差必须按实际分析分辨率或 ROI 尺寸归一化，不能把单一绝对值用于所有设备。
- 曝光至少统计高光裁剪比例、暗部裁剪比例、亮度分位数和答题框/纸面局部对比度；中心点亮度只能保留作诊断，不能单独决定是否接纳。
- 手机连续稳定约 300ms 后，在 400～600ms 候选窗口中逐帧做低成本评估，选择 AF/AE 已接纳且质量分最高的一帧进入完整 OMR，而不是每 300ms 随机取一帧。
- 阈值必须由真机日志和标注帧标定，不能直接恢复未经标定的固定 Laplacian 阈值。建议按设备或分辨率桶记录分布，再确定“明确拒绝”和“可参与择优”两档阈值。
- 质量失败只产生重拍/等待提示，不得保留该帧的答案、分数或共识票数。

### 验收标准

- 模糊、严重过曝和欠曝帧被明确拒绝，不产生错误成功结果。
- 正常室内光照下不会因门槛过严导致扫描长期无响应。
- 质量门控输出卡片 ROI 的关键指标和稳定拒绝码，能够区分对焦未收敛、运动模糊、过曝与欠曝。
- 增加录制相机帧序列的回归测试，并覆盖曝光变化、运动模糊和失焦。

## 3. CameraX 捕获门控已完成，设备能力配置仍未闭环

### 现状

- 分析请求为 1280×960，但允许回退到更低分辨率；旧 L 角标探针在 960 宽时无法定位。
- 分析器已改为低成本逐帧门控，并在 450ms 窗口保留质量最高的一帧进入完整 OMR。
- 已读取 AF/AE 收敛状态、曝光时间和 ISO；固定请求 `[30,30]` AE FPS 仍未按设备能力列表自适应选择。
- Preview 与 ImageAnalysis 没有通过统一 `ViewPort` 绑定，取景与分析区域可能不完全一致。

### 后续方案

- 检查并记录实际分析分辨率；宽度低于 1280 时拒绝扫描或切换更合适的配置。
- 性能允许时评估 1600/1920 宽分析流，并用真实设备测量耗时和热量。
- 配置连续自动对焦和卡片区域测光，并从每帧捕获结果读取 AF state、AE state、曝光时间和 ISO。AF 处于扫描/未对焦状态或 AE 处于搜索/预捕获状态时不进入 OMR；固定焦镜头或设备不提供状态时，按相机能力显式降级到像素质量门控，不能永久等待。
- 将“Camera2 状态已收敛”和“像素质量合格”同时作为接纳条件：AF/AE 收敛不代表图像一定清晰且未过曝，像素指标也不能替代尚在变化的捕获状态。
- 从 Camera2 能力列表选择支持的 FPS range，并使用统一 `UseCaseGroup`/`ViewPort`。
- 保留 `STRATEGY_KEEP_ONLY_LATEST`，将“每 300ms 随机取帧”改为“低成本逐帧评估、择优送入 OMR”。

### 验收标准

- 调试信息包含实际分辨率、AF/AE 状态、曝光时间、ISO、帧时间戳和丢帧原因。
- 预览引导框与分析帧中的卡片区域一致。
- 至少在两台不同相机能力的 Android 设备上验证配置兼容性。

## 4. 旧卡片角标静默错分门控已实现

### 现状

- 新模板默认使用编码角标，1280 宽的轻度模糊/噪声回归结果稳定。
- legacy L 角标在强过曝时会出现 `outer` 与 `innerRight` 映射冲突；当前实现返回 `LEGACY_ANCHOR_AMBIGUOUS`，已知 10/10 变成 8/10 的样例不再输出分数。
- 已打印的旧卡仍必须保留兼容路径，不能直接删除。

### 后续方案

- 新打印卡继续使用编码角标，并在产品提示中逐步引导旧卡换印。
- 新版卡默认使用 `CODED_ONLY` 扫描模式；编码失败时提示对准/重拍，不自动回退到旧卡路径。
- legacy 实心角标或 L 路径出现四角组合、参考候选或映射结果分歧时降级为可见失败，不输出分数。
- 为旧卡增加强过曝、阴影、旋转和运动模糊的专门回归集。

### 验收标准

- 编码路径宁可返回未识别并提示重拍，也不得通过旧实心/L 角回退输出分数。
- legacy 实心/L 路径宁可返回未识别，也不得在候选有分歧或已知强过曝样例上输出错误成功分数。
- 编码角标、实心角标和 L 角标三条兼容路径都有独立的真图回归测试。

## 5. 最急需的优化路径

### P0-1（已完成）：先切断静默错分路径

这是最先实施的改动，因为它直接决定错误分数能否离开扫描器。

1. 为扫描入口增加显式 `AnchorMode`：新版打印/默认扫描使用 `CODED_ONLY`，旧卡入口使用 `LEGACY`。
2. 将定位和读格拆成两个阶段。定位阶段返回带置信诊断的 `AnchorDecision.Accepted`、`Ambiguous` 或 `Rejected`；后两者不能调用读格和评分。
3. 编码路径失败闭合；实心/L 角保留最佳与次佳候选并增加分差、几何一致性和映射一致性门控。
4. 先用合成图覆盖路径锁定、缺失/损坏角标和候选冲突，再用现有真图复现集验证所有失败结果均为 `score=null`。

### P0-2（已完成）：接入 AF/AE 与像素质量双门控

1. 从 CameraX/Camera2 捕获结果取得 AF/AE、曝光时间和 ISO，并连同实际分辨率、时间戳写入每帧诊断。
2. 实现纯 JVM `FrameQualityEvaluator` 及合成模糊、过曝、欠曝测试；先以“明显坏帧拒绝”为目标，避免一开始把正常帧门槛设得过窄。
3. 调整处理顺序为：相机状态门控 → 低成本帧质量 → 角标定位 → 卡片 ROI 质量/歧义门控 → 读格 → 评分 → 多帧共识。
4. UI 显示可操作提示，例如“正在对焦”“光线过强”“画面模糊”“角标不清晰，请重拍”，且旧分数不能在等待/拒绝状态下被误当作本帧新结果。

### P0-3（自动化已完成，真机待补）：真机标定和回归闭环

1. 在至少两台设备上采集同一张编码卡与旧卡的清晰、失焦、运动模糊、过曝、欠曝、阴影和倾斜序列；每帧保留原图、捕获元数据、质量指标、定位候选和最终决策。
2. 建立两类断言：坏帧不得输出分数；清晰帧在合理持稳时间内必须得到正确分数。错误成功的优先级高于识别变慢或拒绝率偏高。
3. 阈值标定完成后再开启候选窗口择优和多机型回归，避免用共识掩盖单帧安全问题。

### P1：相机配置与取景一致性

- 根据设备能力选择分辨率与 FPS，统一 Preview/ImageAnalysis 的 `UseCaseGroup`/`ViewPort`，再评估 1600/1920 宽分析流的性能和热量。
- 这一阶段不能替代 P0 门控；即使相机配置改善，扫描器仍必须对坏帧和歧义定位失败闭合。

## 6. 测试与 Fedora 部署建议

- 纯算法和门控先运行 `sh gradlew :omr-core:test`，再运行编码卡、旧实心卡和 L 角卡的定向真图回归；Android 集成改动运行 `sh gradlew test` 与 `sh gradlew assembleDebug`。
- 可按 `LOCAL_DEVELOPMENT.md` 将工作区同步覆盖到 `fedora:/data/Sources/android-omr` 后测试；远端必须显式设置 JDK 21 与 Android SDK 环境，不能使用默认 Java 8。
- 每次同步前后记录本地与 Fedora 的变更摘要，测试输出至少保留失败用例名、拒绝原因、`anchorMode`/`anchorPath`、AF/AE 状态和质量指标，便于区分代码回归与设备阈值问题。
- 建议按 P0-1、P0-2、P0-3 分批部署。每一批都先验证“错误帧没有分数”，再验证正常帧识别率和延迟，避免同时改定位、相机参数和阈值后无法归因。
