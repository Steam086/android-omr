### 2026-07-10 识别距离与所见即所得整改

#### 根因

- 远距离主瓶颈不是编码角标的 12px 最小候选，而是读格前 `MIN_PROJECTED_CELL_SIZE=14` 与 `MiniProgramBubbleReader.MIN_SAMPLE_SIZE=14` 的双重硬门。以默认带考号的 15 题卡估算，卡片需占 1280 分析帧约 68% 宽度；60 题卡的可用远近比例更窄。
- 原读格器按投影格的自然像素尺寸建立采样矩阵并取整到单个像素，距离变化会同时改变矩阵尺寸、连通域阈值和取样相位，导致同一填涂在不同距离下行为不稳定。
- CameraX Preview 与 ImageAnalysis 原先分别绑定，没有共享 `ViewPort`；预览使用 `FIT_CENTER` 且状态面板覆盖底部，用户看到的构图不是分析器实际裁切范围。
- 近距离时四角中心的 4px 边距门会先拒绝推算出的缺失编码角标；同时缺少独立的“答题格被裁切”拒绝码，用户无法区分该靠近还是远离。
- 完整画面的 Laplacian 模糊硬门会受卡片外背景支配，小卡即使自身清晰也可能在角标定位前被拒绝。

#### P0：所见即所得

- `Preview` 与 `ImageAnalysis` 现在以同一旋转、同一个 `PreviewView.viewPort` 放入一个 `UseCaseGroup`，并在 `PreviewView` 完成布局后绑定。
- 扫描页使用无遮挡的纵屏 3:4 相机区域和按实际模板宽高比计算的推荐框；状态、失败原因和结果卡位于取景画面下方。推荐框只帮助构图，不是另一个隐藏分析 ROI。
- `PreviewView.ScaleType.FILL_CENTER` 与共享 `ViewPort` 配合，CameraX 将相同传感器裁切传给预览和分析；分析适配器继续严格使用 `ImageProxy.cropRect`。

#### P1：扩大远距离范围

- 优先请求 1920×1440，并检查共享裁切后实际 `MiniProgramFrame` 尺寸；低于 1280×960 时返回 `RETAKE_LOW_RESOLUTION`，不调用 OMR、不产生分数，并提示设备相机分析流不兼容（移动距离不能改变该硬件分辨率）。
- 每个投影格先验证至少 10px 宽、8px 高、80px² 面积及 1px 双线性插值边界，再统一重采样为 16×16。0.85 倍合成生产卡已从原来的 `projected cell too small` 变为正确读取 `1234`、指定答案和 10 分；0.7 倍仍安全拒绝。

#### P2：扩大安全近距离范围与符合直觉的操作

## 1. 必须先落实的安全边界

相机扫描的首要目标是避免“看起来成功但分数错误”。只有帧质量合格、角标路径明确、定位候选唯一且几何校验通过的结果才能进入读格、评分和多帧共识。任一条件不满足时都必须返回可区分的失败原因并提示用户重拍，不得生成分数，也不得让失败帧参与共识。

### 编码角标路径

- 新版应用打印的答题卡使用编码角标。扫描会话应明确传入 `CODED_ONLY` 或 `LEGACY` 兼容模式，不能仅凭一次检测失败在三条路径之间自动猜测。
- `CODED_ONLY` 模式下，编码角标无法可靠解码、ID 重复、方向冲突、重投影误差超限或几何校验失败时，直接提示“角标不清晰，请对准后重拍”，不得回退到实心角标或 L 角路径后继续评分。
- 允许由 3 个唯一编码角标推断第 4 个，但必须同时满足解码误码、唯一性、方向、重投影和卡片几何门槛；少于 3 个唯一编码角标不评分。
- 过渡期若尚未增加显式扫描模式，只要出现可信编码角标证据，就必须锁定编码路径；不得把同一帧再解释成旧实心/L 角卡。当前实现仅在检测到至少 2 个编码 ID 时阻止回退，仍不足以覆盖检测到 0～1 个 ID 的新版坏帧。

### 旧实心角标与 L 角路径

- 旧实心角标和 L 角只能在显式 `LEGACY` 兼容模式下启用，且兼容模式也必须失败闭合，不能以“总能给出一个分数”为目标。
- `SolidCornerMarkerDetector` 与 `CornerAnchorMatcher` 必须保留最佳和次佳四角组合的分数、来源和坐标。若两个候选都通过基础校验，但分差不足，或映射后的关键答题格位置不一致，则返回“定位有歧义，请重拍”，禁止输出分数。
- L 角路径还要比较 `outer` 与 `innerRight` 两套参考。若两者证据接近却映射到不同答题格、准考证号或分数，不得按候选顺序静默选一个。
- 不能用多帧多数投票掩盖单帧定位歧义；只有每一帧先独立通过候选唯一性门控，才能进入共识。

### 结果契约与验收标准

- 为拒绝原因使用稳定枚举，例如 `WAIT_FOCUS`、`WAIT_EXPOSURE`、`RETAKE_BLUR`、`RETAKE_EXPOSURE`、`RETAKE_CODED_MARKERS`、`LEGACY_ANCHOR_AMBIGUOUS`，UI 再映射为中文提示；不要依赖英文 `failureReason` 文本判断流程。
- 所有拒绝结果都必须满足 `success=false`、`score=null`，并记录 `anchorMode`、`anchorPath`、最佳/次佳候选分数、分差和最终拒绝原因。
- 编码路径失败不会落入 `solid-marker` 或 `l-bracket`；旧实心/L 角候选有分歧时，不会进入读格、评分或共识。

## 2. 已实现卡片 ROI 质量硬门与整帧软诊断

### 实现现状

- `AndroidOmrImageAnalyzer` 在完整 OMR 前记录设备稳定性、AF/AE 状态和整帧清晰度；稳定性、AF/AE 与整帧模糊只作为诊断和 UI 提示，整帧曝光异常仍是硬门。
- `AndroidOmrEngine` 在定位后对卡片 ROI 再做清晰度、曝光裁剪和动态范围复核；拒绝结果不进入读格、评分或共识。
- 强过曝以及已出现不稳定答案的 `blur2` 真图扰动均回归为可见失败，`score=null`。

### 已实现方案

- 将门控拆成两层：在完整 OMR 前用低成本指标拒绝明显曝光异常并记录模糊 advisory；定位成功后只在卡片四边形及答题区 ROI 内复核清晰度、曝光裁剪比例、局部对比度、卡片面积和重投影误差，然后才允许读格和评分。
- 新增纯 JVM 的 `FrameQualityEvaluator`，返回结构化指标、是否接纳和拒绝原因。Laplacian 方差必须按实际分析分辨率或 ROI 尺寸归一化，不能把单一绝对值用于所有设备。
- 曝光至少统计高光裁剪比例、暗部裁剪比例、亮度分位数和答题框/纸面局部对比度；中心点亮度只能保留作诊断，不能单独决定是否接纳。
- 使用 `STRATEGY_KEEP_ONLY_LATEST`，最多每 250ms 转换并分析一个最新帧；不再等待额外候选窗口，也不因手持运动、AF 扫描或 AE 搜索反复清空识别进度。
- 阈值必须由真机日志和标注帧标定，不能直接恢复未经标定的固定 Laplacian 阈值。建议按设备或分辨率桶记录分布，再确定“明确拒绝”和“可参与择优”两档阈值。
- 质量失败只产生重拍/等待提示，不得保留该帧的答案、分数或共识票数。

### 验收标准

- 模糊、严重过曝和欠曝帧被明确拒绝，不产生错误成功结果。
- 正常室内光照下不会因门槛过严导致扫描长期无响应。
- 质量门控输出卡片 ROI 的关键指标和稳定拒绝码，能够区分运动模糊、过曝与欠曝；AF/AE 状态单独作为软信号记录。
- 增加录制相机帧序列的回归测试，并覆盖曝光变化、运动模糊和失焦。

## 5. 最急需的优化路径

### P0-1（已完成）：先切断静默错分路径

这是最先实施的改动，因为它直接决定错误分数能否离开扫描器。

1. 为扫描入口增加显式 `AnchorMode`：新版打印/默认扫描使用 `CODED_ONLY`，旧卡入口使用 `LEGACY`。
2. 将定位和读格拆成两个阶段。定位阶段返回带置信诊断的 `AnchorDecision.Accepted`、`Ambiguous` 或 `Rejected`；后两者不能调用读格和评分。
3. 编码路径失败闭合；实心/L 角保留最佳与次佳候选并增加分差、几何一致性和映射一致性门控。
4. 先用合成图覆盖路径锁定、缺失/损坏角标和候选冲突，再用现有真图复现集验证所有失败结果均为 `score=null`。

### P0-2（已完成）：接入 AF/AE 软信号与像素质量硬门

1. 从 CameraX/Camera2 捕获结果取得 AF/AE、曝光时间和 ISO，并连同实际分辨率、时间戳写入每帧诊断。
2. 实现纯 JVM `FrameQualityEvaluator` 及合成模糊、过曝、欠曝测试；先以“明显坏帧拒绝”为目标，避免一开始把正常帧门槛设得过窄。
3. 调整处理顺序为：250ms 节流 → 相机状态诊断 → 低成本帧质量硬门 → 角标定位 → 卡片 ROI 质量/歧义门控 → 读格 → 评分 → 3/4 多帧共识。
4. UI 显示可操作提示，例如“请持稳手机”“光线过强”“画面模糊”“角标不清晰，请重拍”，且旧分数不能在等待/拒绝状态下被误当作本帧新结果。

### P0-3（自动化已完成，真机待补）：真机标定和回归闭环

1. 在至少两台设备上采集同一张编码卡与旧卡的清晰、失焦、运动模糊、过曝、欠曝、阴影和倾斜序列；每帧保留原图、捕获元数据、质量指标、定位候选和最终决策。
2. 建立两类断言：坏帧不得输出分数；清晰帧在合理持稳时间内必须得到正确分数。错误成功的优先级高于识别变慢或拒绝率偏高。
3. 真机记录从卡片完整进入画面到锁定的 P50/P95；P95≤1.5 秒为目标，同时验证 3/4 共识没有掩盖单帧安全问题。

### P1（代码已完成，真机性能待验）：相机配置与取景一致性

- Preview/ImageAnalysis 的统一 `UseCaseGroup`/`ViewPort`、1920×1440 优先请求和最低实际分辨率契约已完成；根据设备能力选择 FPS、评估 1920 宽分析流性能和热量仍待真机。
- 这一阶段不能替代 P0 门控；即使相机配置改善，扫描器仍必须对坏帧和歧义定位失败闭合。

## 6. 测试与 Fedora 部署建议

- 纯算法和门控先运行 `sh gradlew :omr-core:test`，再运行编码卡、旧实心卡和 L 角卡的定向真图回归；Android 集成改动运行 `sh gradlew test` 与 `sh gradlew assembleDebug`。
- 可按 `LOCAL_DEVELOPMENT.md` 将工作区同步覆盖到 `fedora:/data/Sources/android-omr` 后测试；远端必须显式设置 JDK 21 与 Android SDK 环境，不能使用默认 Java 8。
- 每次同步前后记录本地与 Fedora 的变更摘要，测试输出至少保留失败用例名、拒绝原因、`anchorMode`/`anchorPath`、AF/AE 状态和质量指标，便于区分代码回归与设备阈值问题。
- 建议按 P0-1、P0-2、P0-3 分批部署。每一批都先验证“错误帧没有分数”，再验证正常帧识别率和延迟，避免同时改定位、相机参数和阈值后无法归因。
